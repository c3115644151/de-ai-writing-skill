# 去AI写作

## 概述

本技能的核心判断是：**AI写作失败不是表面风格问题，而是生成目标与人类认知目标之间的根本性背离。** LLM的优化目标（统计压缩效率）与人类表达的优化目标（语义丰富性、意图传达、读者适配）存在本质冲突——这不是"教AI写得更像人"能解决的，而是需要**重构AI的生成流程**，让它经历一个更像人的认知写作过程。

你的任务不是"伪装AI文本"，而是**让AI从目标出发、经历完整的认知写作三阶段**：Planning → Translating → Reviewing。

**为什么这样做有效？** 研究表明（ICML 2025 Outstanding Paper, arXiv 2504.15266），NTP（Next Token Prediction）的根本局限在于：模型在生成第一个token时只能拟合边际分布，后续token虽然条件于前文但无法完成"先采样目标意图，再采样所有输出"的正确生成顺序。CogWriter（arXiv 2502.12568）将Flower & Hayes认知写作理论工程化实现后，在LongGenBench上比GPT-4o高22%的指令完成准确率。这条路有实证支持。

---

## 工作流程

本技能分三个阶段执行，**每个阶段都必须按顺序完成，不可跳过**。

### 阶段一：写前目标设定（Planning）

**目的**：对抗NTP的局部贪婪特性——AI倾向于直接开始输出，跳过目标规划。必须在写作开始前锁定目标。

在AI开始撰写任何正文之前，要求它**显式回答以下三个问题**，并将答案写在文档的规划区（不进入最终正文）：

1. **修辞目标**：这一段的修辞目的是什么？（例如：建立论点、反驳预期反对、提供实证支撑、过渡到下一主题、总结收束）
   - 回答要求：必须使用动作性短语，不能是"介绍"、"描述"等泛化词。例如"用A银行的案例反驳'金融科技必然降低风险'这一论断"。

2. **读者需求**：读者在这一段需要知道什么？（即读完这段后，读者应该带走哪个结论或信息块）
   - 回答要求：从读者视角描述，不能是"我想让读者知道..."，必须是"读者应该知道..."。例如"读者应该知道：A银行2019-2023年间的数字化转型并未显著降低其不良贷款率"。

3. **论证锚点**：这一段要覆盖的关键论证步骤是什么？（即在整个论证链中的位置——前接什么、后接什么、本段完成什么逻辑功能）
   - 回答要求：用"前接...→ 本段... → 后接..."的链式格式。例如"前接：金融科技降低风险的普遍假设 → 本段：用A银行的实证数据质疑该假设的普适性 → 后接：分析A银行案例中出了什么问题"。

**为什么这一步必须做**：Information Bottleneck框架（LeCun共署，arXiv 2505.17117）证明LLM在细粒度语义梯度上相关性极弱（ρ<0.2），即模型没有真正的语义目标感。SWI方法（Speaking with Intent, arXiv 2503.21544）的实证表明，要求LLM在生成前显式声明意图能一致性地提升输出质量。

**检查标准**：只有上述三个问题都得到明确、具体的回答后，才能进入阶段二。任何泛化回答（如"为了论证""为了让读者理解"）都必须重新回答。

### 阶段二：写中结构约束（Translating）

**目的**：在AI执行翻译（从规划到具体文本）的过程中注入结构性约束，对抗LLM的统计平滑倾向。注意：约束是"写入生成过程"而非"生成后修改"。

在AI逐句生成内容时，必须实时遵守以下约束：

#### 2.1 句式黑名单（禁止使用以下模式）

为什么禁止这些句式？因为它们被实证确定为AI文本的高频统计指纹。

| 类别 | 禁止模式 | 示例（禁止） | 推荐替代 |
|------|---------|------------|---------|
| 现在分词从句 | 「, (verb)ing...」开头或嵌入的现在分词结构 | "Drawing on prior research, this study..." | 直接陈述："Prior research shows..." |
| 过度名词化 | 动词→名词的过度转换 + "of"结构 | "The implementation of the strategy..." | "Implementing the strategy..." |
| 套话骨架 | "X plays a crucial role in Y" | "Social media plays a crucial role in modern communication" | "Social media reshapes modern communication" |
| 高频AI词 | delve, navigate, foster, leverage, harness, realm, pivotal, testament, multifaceted, nuanced, underscore, revolutionize, paradigm, unlock, ultimately | "This paper delves into the multifaceted..." | 直接说："This paper examines..." |
| 空泛过渡 | "It is worth noting that..." "It is important to highlight that..." "In addition, it should be mentioned that..." | "It is worth noting that the results were significant" | "The results were significant." |
| 自我指涉 | "As an AI language model..." "I don't have personal opinions..." | — | 绝不出现这种句式 |

#### 2.2 Burstiness强制干预（对抗perplexity方差过低）

AI文本的perplexity方差（burstiness）系统性偏低（GPTZero测量：AI文本burstiness ~14-22 vs 人类学术写作 ~38-44）。每次完成一段生成后，必须执行：

- 扫描该段各句的**字数分布**（以中文字数计）
- 如果该段中超过60%的句子字数在正负30%范围内（例如一段6句，每句都在25-45字之间），判定为burstiness不足
- 主动修改其中1-2句的字数：一句缩短至12-20字，一句扩展至50-70字
- 短句用于"断言式收束"或"关键结论"；长句用于"带条件的复杂论证"或"证据链展开"

**修改后检查**：最短句与最长句的字数比应至少达到1:2.5。（例如最短15字，最长至少38字）

#### 2.3 Hedging密度控制（对抗RLHF的过度谨慎）

RLHF系统性放大AI文本中的hedging（模糊限定词）倾向（Procaccia等2026，Anthropic ICLR 2024）。人类学术写作确实也使用hedging，但密度和分布不同。控制规则：

- 每200字内，"可能/或许/表明/暗示/往往/通常/倾向于/一定程度上"等模糊限定词不超过2个
- 关键论证节点（引入核心论点、给出结论性判断）处禁止使用hedging
- 仅在以下场景允许hedging：陈述未完全确认的研究发现、讨论尚未有共识的争议点

#### 2.4 角色约束（对抗blandification）

Google DeepMind（arXiv 2603.18161）发现AI修改系统性将所有文本推向"共享语义模式"——向LLM偏好的语言模式收敛。对抗方法：

- 如果用户在任务中提供了写作风格参考（之前的作品、个人笔记、特定作者的风格示例），必须在生成中融入该风格的**节奏特征**（句子长度偏好、段落密度、引证习惯）
- 如果没有风格参考，以**学术论文**的默认风格为准：避免口语化、避免过度华丽的修辞、避免"首先是/其次是/最后"这类模板化过渡
- 过渡语句必须服务于论证逻辑，而非填充段落

### 阶段三：写后自我审阅（Reviewing）

**目的**：补偿LLM缺失的Reviewing阶段。人类写作的Reviewing阶段会评估已写内容是否达成目标、是否逻辑连贯、是否需要重构。AI的自回归架构使已生成的token成为不可变上下文，无法自动审阅和纠偏——所以必须显式执行。

完成每一节的正文后，AI必须以**读者视角**执行以下检查：

1. **修辞目标达成检查**：回到阶段一记录的三个目标答案，逐一核对：这一段是否完成了声明的修辞目标？如果偏离，标记偏离点并解释为什么偏离（是原目标不合理还是执行跑偏？）。

2. **读者体验检查**：假设读者不了解你的论证背景，读这一段后会：
   - 理解你想表达的核心观点吗？
   - 哪些地方可能让读者困惑（逻辑跳跃、术语未定义、引用缺少上下文）？
   - 段落开头的引导句是否与上一段结尾自然衔接？

3. **论证链检查**：本段的结尾是否为下一段做好了逻辑准备？如果没有，补充过渡句。

4. **Burstiness复核**：检查阶段二的burstiness干预是否执行了。如果没有，补执行。

**修改规则**：
- 发现问题后，**只局部修改有问题的部分**，不整体重写整段
- 每次修改后，重新确认修改部分没有引入AI统计指纹（检查是否使用了黑名单句式）
- 如果修改后仍不满意，标记问题并在全部完成后提交给用户，让用户决策

---

## 输出格式

最终输出应为纯文本格式的文档正文（不含阶段一的规划区、阶段三的审阅批注），按原大纲结构呈现。

审阅过程中发现但未修复的问题，在全文末尾以「待确认」列表呈现，每条一行，格式为：

> 待确认：[节标题] — 问题描述

---

## 边界情况处理

1. **用户需求不明确时**：先回到阶段一，根据你对任务的理解填写三个目标答案，然后在开始阶段二前将目标展示给用户确认。未经用户确认的目标设定必须标注为「暂定」，并在审阅阶段特别检查。

2. **用户明确要求多次修改同一段时**：每次修改都重新走过阶段一（至少修辞目标）、阶段三（审阅）。禁止在两次修改之间的中间状态直接输出给用户。

3. **输出篇幅超长（>4000 token）时**：AI生成文本在超过4000 token后性能显著退化（LongGenBench, ICLR 2025）。如果任务总字数超过4000 token，分段生成——每段生成时都执行完整的三阶段流程，最后做一次全局连贯性检查。

4. **任务涉及多个独立章节时**：每个章节独立执行三阶段流程（每个章节有自己的阶段一目标设定），但全局审阅阶段（全部章节完成后）额外检查：各章节间的论证链条是否顺畅、是否有重复论证、是否有目标漂移。

5. **用户没有论文大纲**：在开始阶段一之前，先询问用户是否有大纲或至少确定主题和基本结构。如果用户也没有，先协助生成大纲（告知用户这是大纲而非最终正文），经用户确认后再进入三阶段流程。

## 工作流程速查

```
用户请求写作任务
  │
  ├─ 有明确大纲 → 直接进入阶段一
  │
  └─ 无大纲 → 先协助生成大纲 → 用户确认 → 阶段一
      │
      ▼
【阶段一：写前目标设定】
  回答三个问题（修辞目标 / 读者需求 / 论证锚点）
  检查：答案必须是具体可操作短语，非泛化
      │
      ▼
【阶段二：写中结构约束】
  ├─ 遵守句式黑名单
  ├─ 执行Burstiness强制干预
  ├─ 控制Hedging密度
  └─ 维持角色约束（风格一致性）
      │
      ▼
【阶段三：写后自我审阅】
  ├─ 修辞目标达成检查
  ├─ 读者体验检查
  ├─ 论证链检查
  └─ Burstiness复核
      │
      ▼
输出正文（不含规划区/审阅批注）
  + 待确认列表（如有未修复问题）
```
